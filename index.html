Intent Routing

先分流，
再决定谁来回答

用户输入先经过纯分类网关，不直接生成回复，而是分流到问答、导购、订单和兜底分支，避免一个模型同时承担分类和回答。

LLM · GATEWAY 意图分流助手
GREETING — 打招呼
ABUSE — 辱骂兜底
END — 道别收束
SHOPPING — 导购推荐
BUSINESS — 业务问答
分流模型 deepseek-v4-flash · 快、省 token，专职做选择题
Query Rewriting

听懂"那XX呢"

这一层处理真实对话里的省略句、代称和预算变化。补全之后再进入检索和导购分支，提升后续参数提取和回答稳定性。

Redis Cache & Concurrency Control

不该重复的事，
一次都不多做

重复问题直接命中缓存，短时间重复提交由互斥锁拦截，减少上游模型调用，并控制并发下的重复穿透。

cache_proxy.pycache hit / lock guard
def make_cache_key(query: str) -> str:
    query_hash = hashlib.md5(query.strip().lower().encode("utf-8")).hexdigest()
    return f"dify_cache:{query_hash}"

cached = redis_client.get(cache_key)
if cached:
    return {"source": "cache", "answer": json.loads(cached)["answer"]}

if redis_client.exists(lock_key):
    return {"source": "lock_wait", "answer": "您的问题正在处理中，请稍候..."}
