# TunTunAgent

TunTunAgent 是一个面向高频售前咨询场景的 AI 系统工程化展示项目，重点展示 Dify 工作流、FastAPI 业务接入层、Redis 缓存保护和压测验证如何组合成一条可部署、可解释、可演示的完整链路。

## 项目地址

- 展示页：`http://101.43.56.2:8888`
- 在线体验：通过展示页中的“立即体验”按钮跳转

## 核心特点

- 将 Dify 作为 AI 工作流编排层，而不是直接把官方聊天页当作唯一交付形态
- 用 FastAPI 补齐真实业务接入层，承接导购接口调用、缓存代理和工程控制逻辑
- 用 Redis 实现重复问题缓存、互斥锁保护和缓存管理接口，降低大模型重复调用成本
- 将“知识问答”和“商品导购”拆成两条链路，避免该查数据的问题被模型自由发挥
- 对模型输出做清洗和兜底，避免思考标签、异常内容或低质量回答直接暴露给用户

## 技术栈

## 架构设计

### 1. AI 编排层与业务接入层拆分

项目没有把全部逻辑都堆进一个 Prompt，而是明确分成两层：

- Dify 负责意图分流、知识库问答、导购推荐、异常分支编排
- FastAPI 负责外部接口接入、缓存代理、并发保护和工程增强

这部分明确了系统边界：Dify 负责理解和编排，FastAPI 负责确定性接口接入和工程控制，避免把全部逻辑塞进单一 Prompt。

### 2. 多意图分流

系统先做分类，再进入对应链路，避免一个模型同时承担分类、检索、生成和纠错。

当前公开展示版覆盖的主场景包括：

- 咨询问答
- 商品导购
- 订单查询
- 售后引导
- 打招呼 / 结束语 / 异常辱骂

### 3. 知识问答与导购链路分治

- 知识类问题走 RAG 检索 + 重排序
- 导购类问题走外部商品检索接口 + 价格/皮肤筛选逻辑

这样可以把“可检索的确定性数据”和“模型生成能力”分层处理，减少幻觉和乱答。

### 4. Redis 缓存与互斥锁保护

围绕高频重复问题，项目额外实现了 `cache_proxy.py`：

- 首次请求：回源 Dify，写入 Redis
- 重复请求：直接命中缓存
- 转人工类回答：不缓存
- 高频并发下：通过互斥锁避免同一问题短时间内反复打穿上游

同时提供了简单的缓存管理接口，便于查看缓存状态和手动清理。

## 关键工程点

### 1. 对话改写与参数补全

在正式进入分类和导购前，系统先对用户表达做轻量补全，处理省略、代称、预算区间变化等口语化输入，提升后续分类和导购参数提取稳定性。

### 2. 检索质量控制

知识库召回曾出现过因分段方式导致专有名词丢失的问题，后续改为保留原文结构的分块方式，并叠加 `qwen3-rerank` 做重排序，阈值低于标准时宁可不答。

### 3. 输出清洗与兜底

对 DeepSeek 输出做二次清洗，避免把不适合用户看到的思考标签直接返回给前端；同时对知识库无结果、导购无匹配、接口异常、模型异常等情况都设计了兜底逻辑。

### 4. 展示链路与工程链路分离

当前项目同时保留了两条可演示链路：

- 展示链路：展示页 `8888` -> Dify 会话页
- 工程链路：业务接口 / 缓存代理 -> Dify API

这样既能保留 Dify 页面用于快速演示，也能展示真实工程接入能力。

## 实测结果

### 1. 导购接口压测

对商品检索接口 `search_accounts` 做过 50 并发压测，结果如下：

- 成功率：100%
- 平均延迟：0.206s
- P95：0.214s
- P99：0.219s
- 最大延迟：1.071s

### 2. RAG / FAQ 问答链路压测

对 Dify 聊天 API 做过真实链路验证，选取不依赖外部导购 HTTP 节点的 FAQ 问题“押金怎么算的”：

- 1 并发 1 请求：成功率 100%，单次约 16.3s
- 10 并发 10 请求：成功率 100%，平均延迟约 14.748s
- 20 并发 20 请求：成功率 95%，平均延迟约 21.775s，出现 1 次 30s 超时

这组数据对应的是真实工作流链路耗时，而不是本地 mock 或静态演示。10 并发可以视为当前轻量云服务器上的稳定基线，20 并发更接近资源边界验证。

### 3. 导购 / HTTP 检索链路验证

针对会调用外部 HTTP 商品检索节点的导购问题，也做过链路验证：

- 问题示例：`500以内有信条号吗`
- 10 并发 10 请求：成功率 100%，平均延迟约 22.265s

这说明复杂业务链路在引入外部检索后，整体响应耗时会显著高于纯 FAQ / RAG 问答链路。

### 4. Redis 缓存命中验证

以“押金怎么算的”为例：

- 首次请求：`source = dify_live`
- 重复请求：`source = cache`

说明缓存代理链路已实测生效，重复问题可以跳过上游大模型调用，直接从 Redis 返回结果。

### 5. 性能结论

- 导购检索接口本身可以稳定承载高并发
- FAQ / RAG 问答链路在 10 并发下表现稳定
- 在 4 核 4G 的轻量云服务器上，完整 AI 链路在 20 并发时已接近资源上限，开始出现超时
- 因此项目额外实现了 Redis 缓存代理和业务接入层，用于优化重复问题、保护上游工作流并为正式业务接入预留统一控制面
